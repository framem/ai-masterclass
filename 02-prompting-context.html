Warum überhaupt Prompting & Context?

Der Prompt ist die Bestellung.
Der Context ist die Küche.

Ein Modell arbeitet nur mit dem, was es bekommt. Vage Bestellung · planlose Küche = beliebige Antwort.
Klare Bestellung · gut vorbereitete Küche = verlässliches Werkzeug.

Merksatz
Ein guter Prompt ist nicht länger, sondern spezifischer. Ein guter Context ist nicht größer, sondern gezielter.

Patterns > Tricks

Fünf Muster, die wirklich ziehen.

Jedes einzeln einsetzbar — und kombinierbar, messbar, wiederverwendbar.

01
Few-Shot · zeigen

1–3 Input/Output-Paare. Der mit Abstand stärkste Hebel — wenn man nur einen nutzt: diesen.

in: „Logout 500 bei abgelaufenem Token"
out: {"area":"auth","sev":"high","kind":"bug"}
02
Output-Format · Gefäß

XML-Tags, JSON-Schema, Markdown-Struktur — gibt dem Modell ein klares Gefäß zum Hineinschreiben.

<analysis>…</analysis>
<answer>…max 3 bullets…</answer>
03
Role · System-Prompt

Macht aus generisch jemand Bestimmtem: Expertise, Ton, Do/Don't.

„Du bist Senior Tech-Lead, 10 J. Erfahrung mit verteilten Systemen. Knapp, technisch, kein Hype."
04
Constraint · Grenzen

Länge, Vokabular, erlaubte Werte. Begrenzt ausufernde Antworten. Negative Constraints („nutze nicht…") sind möglich, positive sind besser.

„Max 3 Bullet-Points, je ≤ 15 Wörter.
Nur Werte aus: low | med | high."
05
Self-Check · zweite Iteration in derselben Antwort

Lass das Modell vor der finalen Antwort einmal prüfen, ob es alle Anforderungen erfüllt. Geringer Aufwand, aber wirkungsvoll bei komplexen Outputs.

„Bevor du antwortest, prüfe in <check>: 1) hält Längen-Constraint? 2) alle 3 Felder gesetzt? 3) keine Marketing-Phrasen?
Dann gib die finale Antwort in <answer>."

Context Engineering

Context ist das Arbeitsgedächtnis des Modells.

Alles, was rein soll, kostet Tokens. Token-Budget ist endlich — und je voller das Fenster, desto schlechter die Antwortqualität. Just-in-time schlägt just-in-case.

Echte Praxis — vager Prompt → produktionsreifer Prompt

Refactored einen schlechten Prompt in drei Iterationen.

15 Min Refactoren · 5 Min Debrief. Driver tippt, Navigator ist „Prompt-Doctor" — fragt, blockt, verbessert.

Setup · 2 min 1 Paarbildung 2 Vagen Prompt unten links wählen 3 Test-Input wählen · z. B. eine echte Kundenzeile 4 Timer 15:00

Konkret: eine AGENTS.md, die funktioniert

So sieht eine gute AGENTS.md aus.

AGENTS.md ~480 tokens · TS/Node-Repo 
# Acme API

## Stack
Node 22 · TypeScript strict · Fastify · Postgres · Vitest

## Commands
- pnpm dev       — local server (port 3000)
- pnpm test      — vitest, must be green before commit
- pnpm lint      — eslint + prettier, no warnings allowed
- pnpm db:migrate — drizzle migrations

## Style
- tabs, not spaces (legacy reasons, do not "fix")
- error classes in src/errors/, NEVER throw plain Error
- prefer Result<T,E> over try/catch in business logic
- file size cap: 300 LOC; split otherwise

## Hard rules
- NEVER edit migrations/ after merge — add new migration
- NEVER call external APIs from src/domain/ — stays pure
- ALL routes go through withRateLimit()

## Conventions you can assume
- vitest BDD style, describe/it, no jest globals
- env vars validated in src/config/env.ts
+ Gehört rein
  • Stack & wichtigste Commands
  • Repo-spezifische Konventionen
  • Harte Regeln („NIE direkt Postgres-Connection")
  • Annahmen, die ein neuer Dev wissen müsste
− Gehört NICHT rein
  • Onboarding-Roman, Firmen-Vision
  • Code-Beispiele (→ Skill oder README)
  • Daten, die sich oft ändern (Versionen, URLs)
  • Persönliche Vorlieben ohne Begründung
Faustregel
Knapp, repo-spezifisch, kein Onboarding-Roman. Stil, Commands, harte Regeln — alles, was der Agent bei jeder Nachricht braucht, und nichts mehr.
Merksatz
Eine AGENTS.md ist kein Wiki, sondern was der Agent bei jeder Nachricht wissen muss. Alles andere gehört nach Skill, Memory oder MCP.
