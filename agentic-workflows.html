Warum überhaupt ein Workflow?

Ein Prompt erzeugt Text.
Ein Workflow erzeugt Ergebnisse.

Sobald ein Modell handeln soll — Dateien lesen, Code ändern, Tests laufen lassen — reicht der einzelne Prompt nicht mehr. Es braucht eine wiederholbare Choreografie: was geschieht in welcher Phase, mit welcher Information, unter welcher Kontrolle.

Naiv · One-Shot
„Mach das halt."
unstrukturiert
user  „Bau mir Feature X."
agent  schreibt sofort Code, ohne den Rest zu kennen
  • erfindet Konventionen, statt sie zu lesen
  • duplikate Funktionen, Stilbrüche
  • kein Punkt, an dem ein Mensch eingreifen kann
Strukturiert · Workflow
Erst verstehen,
dann handeln.
phasiert
01  Code & Kontext lesen
02  Plan vorschlagen, freigeben lassen
03  klein implementieren, prüfen
04  sauber committen
  • jede Phase erzeugt etwas Prüfbares
  • Mensch greift gezielt an Übergängen ein
  • wiederholbar über Teams und Aufgaben hinweg
Merksatz
Agentisch arbeiten heißt nicht „Agent macht alles allein". Es heißt: klare Phasen, klare Übergaben — der Agent füllt die Phasen, der Mensch öffnet die Tore.

Abgrenzung

Workflow oder Agent?

Beide lassen ein Modell handeln — der Unterschied liegt darin, wer den Pfad bestimmt.

Workflow
fester Pfad
Das Modell füllt die Schritte.

Phasen und Übergänge stehen vorher fest. Das Modell entscheidet nicht über die Struktur, nur über den Inhalt jeder Phase.

  • vorhersagbar & wiederholbar
  • prüfbar an jedem Übergang
  • der Mensch öffnet die Tore
Agent
freie Wahl
Das Modell wählt den Pfad.

Nur das Ziel steht fest. Das Modell entscheidet zur Laufzeit selbst, welche Werkzeuge es in welcher Reihenfolge aufruft — und läuft in einer Schleife, bis das Ziel erreicht ist.

  • flexibel, kann verzweigen & wiederholen
  • weniger vorhersagbar
  • steuert die eigene Schleife
Brücke
Explore → Plan → Execute → Commit ist beides zugleich: ein Agent, der einem Workflow folgt. Die Autonomie des Agenten bekommt die Leitplanken fester Phasen.

Warum gerade diese vier Phasen?

Jede Phase schützt vor einem
konkreten Fehlermuster.

Wer eine Phase überspringt, bekommt zuverlässig den dazugehörigen Fehler. Deshalb ist die Reihenfolge selbst die wichtigste Schutzmaßnahme.

schützt vor →
Halluzinierten Konventionen

Ohne Lesen erfindet der Agent Pfade, Namen und Patterns, die im Repo gar nicht existieren — Code, der für sich genommen nach Lehrbuch aussieht und im Projekt komplett aus dem Rahmen fällt.

// Anti-Pattern:
„Ich nehme an, ihr macht das mit einer Standard-Vorlage…"
Plan
schützt vor →
Falsche Aufgabe gelöst

Ohne expliziten Plan wird der erste Aspekt der Frage gelöst, der dem Modell ins Auge sticht — selten der wichtigste. Der Plan macht Annahmen sichtbar, bevor Code sie zementiert.

// Anti-Pattern:
500 Zeilen Code zur Frage, ob es überhaupt der richtige Ort ist.
Execute
schützt vor →
Theorie ohne Realität

Pläne, die nie laufen, lassen sich nie überprüfen. Erst die Ausführung — Schritt für Schritt, mit Tests dazwischen — zeigt, was wirklich greift. Kleine Iterationen schlagen den großen Wurf.

// Anti-Pattern:
„Im Prinzip funktioniert das so…" — ohne es je auszuprobieren.
Commit
schützt vor →
Verlorener Arbeit

Ohne Commit ist nichts reproduzierbar, nichts review-fähig, nichts wieder einzuspielen, wenn der nächste Versuch schiefgeht. Der Commit gibt dem Menschen den natürlichen Stop-and-Check-Punkt.

Was den Workflow am Leben hält

Drei Hebel, drei Stolperfallen.

+ Hebel
Was den Workflow trägt
Aufgaben klein schneiden

Ein Workflow pro Outcome, nicht pro Feature-Epic. Lieber zehn kleine Loops als ein heroischer.

Plan explizit absegnen

Das Gate zwischen Plan und Code ist der billigste Punkt, um eine falsche Richtung zu stoppen.

Häufig committen

Jeder grüne Zwischenstand ist ein Speicherpunkt. Ein Commit alle 20 Minuten ist normal, kein Overhead.

− Stolperfallen
Was den Workflow zerbricht
Plan im Kopf des Modells lassen

Ein Plan, der nicht aufgeschrieben ist, ist kein Plan. Er kann nicht geprüft, nicht verbessert, nicht erinnert werden.

Mitten in Execute die Aufgabe ändern

Neue Anforderung = zurück zu Plan. Nicht im laufenden Loop dranhängen — sonst verwässert der Kontext.

Den Agent „einfach machen" lassen

Ohne Gate kein Workflow — nur ein langer Monolog mit Schreibrechten.

Take-away
Explore → Plan → Execute → Commit ist kein Prozess-Overhead, sondern ein Vertrag: der Agent verspricht, in dieser Reihenfolge zu arbeiten — der Mensch verspricht, an den richtigen Stellen einzugreifen.

EPEC an die Aufgabe anpassen

Ein Grundmuster, drei Spielarten.

Das Skelett bleibt gleich. Was sich ändert, ist das Artefakt, das den Inner-Loop steuert: ein roter Test, ein Zielbild — oder der Plan selbst.

VARIANTE 01
verifizierbar
TDD-Loop
der Test ist das Ziel
Explore → Plan →
Tests schreiben (rot) → Commit →
Code, bis grün → Commit

Erst Tests schreiben, die bewusst fehlschlagen — und committen. Dann implementieren, bis alles grün ist. Der Agent jagt ein hartes, prüfbares Ziel statt einer vagen Beschreibung.

Wann
Logik mit klaren Erwartungen · Bugfixes (Regression!) · Refactorings
VARIANTE 02
sichtbar
Visual-Loop
das Mockup ist der Maßstab
Explore → Plan →
Code → Screenshot → vergleichen →
Code → … → Commit

Mock oder Screenshot als Zielbild mitgeben. Der Agent baut, macht selbst einen Screenshot, vergleicht — und iteriert. 2–3 Runden sind normal, die erste Version ist selten die beste.

Wann
Frontends · Reports · Diagramme — alles mit sichtbarem Output
VARIANTE 03
volle Kontrolle
Plan-only
der Mensch übernimmt Execute
Explore → Plan →
✋ Übergabe an den Menschen →
selbst implementieren

Der Agent liest und plant — getippt wird selbst. Die billigste Variante, fremden Code zu verstehen, und die sicherste, wenn jede Zeile zählt.

Wann
Security-kritischer Code · Onboarding in fremde Codebases · Architektur-Fragen
Merksatz
Die Reihenfolge Explore → Plan → Execute → Commit ist nie verhandelbar — das Artefakt schon. Gib dem Agenten das härteste Ziel, das die Aufgabe hergibt: Test > Zielbild > Beschreibung.

Wenn der Loop entgleist

Vier Stufen der Kurskorrektur.

Agenten laufen nicht fehlerfrei — das ist eingeplant. Entscheidend ist, früh und billig zu korrigieren, statt zuzuschauen. Von sanft nach hart:

01
Unterbrechen
Signal: falsche Richtung, aber nah dran
Stoppen (Esc), Anweisung nachschärfen, weiterlaufen lassen — der Kontext bleibt erhalten.
kostet: Sekunden
02
Zurückspulen
Signal: der Prompt selbst war das Problem
Zum letzten Checkpoint zurück, Prompt präziser formulieren, neu ansetzen — statt auf einer schiefen Antwort weiterzubauen.
kostet: Minuten
03
Kontext leeren
Signal: der Verlauf ist überladen und unübersichtlich
Stand kurz zusammenfassen lassen → /clear → mit der Zusammenfassung frisch starten. Alte Irrwege vergiften sonst jede neue Antwort.
kostet: einen Neustart
04
Hard Reset
Signal: das Ergebnis selbst ist verbaut
Zurück zum letzten funktionierenden Stand, neuen Loop bei Plan beginnen — nicht im Trümmerfeld weiterflicken.
kostet: einen Loop
Heuristik
Zweimal nachgebessert und immer noch falsch? Eine Stufe eskalieren. Sunk-Cost gilt auch für Prompts — und regelmäßiges Zwischenspeichern ist der Grund, warum Stufe 4 billig ist.

Echte Praxis — im Pair den EPEC-Loop fahren

Macht einen EPEC-Loop — zu zweit, Tor für Tor.

15 Min Coden · 5 Min Debrief. Einer schreibt, einer moderiert — und stoppt den User, wenn er eine Phase überspringen will.

Setup · 2 min 1 Paarbildung 2 Task unten links wählen 3 User tippt den ersten Prompt 4 Navigator nennt das erste Tor (Explore) & prüft 5 Timer 15:00
Rolle · User
User
am Laptop · tippt
Öffnet ein Repo + Coding-Agent. Sagt laut, was passiert. Überspringt keine Phasen im Alleingang.
Rolle · Navigator
Navigator
setzt EPEC durch · moderiert
Beobachtet, fragt, blockt. Darf nicht selbst tippen. Setzt die 4 Tore rechts durch — ist die wichtigste Person im Pair.
Pro Runde
Navigator nennt das nächste Tor → User geht zum Folgeschritt (Explore → Plan → Code → Commit) → Navigator prüft, bevor es weitergeht → wiederholen, bis der Task durch alle 4 Tore ist.
Spielregel · die 4 Tore des Navigators
Diese vier Sätze fallen laut.
01
Explore: „Was hast du gelesen?" → Verlange eine kurze Explore-Zusammenfassung.
02
Plan: „Zeig mir den Plan, schriftlich." → Keine Implementierung ohne nummerierte To-Dos.
03
Execute: „Lass die Tests laufen — nicht raten." → „Im Prinzip funktioniert das" zählt nicht.
04
Commit: „Commit jetzt — mit guter Message." → type(scope): Warum, nicht nur Was.
Tasks · wählt einen
In 15 Minuten realistisch
A
Logging-Middleware: HTTP-Logger mit Method · Path · Status · Latency. Test auf 500er.
B
README „Quickstart in 60s": drei klar-getrennte Schritte für ein bestehendes Repo, copy-pastable.
C
ENV-Validation beim Boot: Pflicht-Vars prüfen, hartes Fail mit klarer Fehlermeldung.
D
Test-Lücke schließen: einen offensichtlich ungetesteten Edge-Case mit gutem Test absichern.
E
Eigener Task aus dem aktuellen Repo — z. B. fehlende Übersetzungs-Keys finden & ergänzen.
Debrief · 5 min in der Runde
Drei Fragen
  • Welche Phase war am schwierigsten?
  • Wolltet ihr eine Phase überspringen?
  • Hat eure Aufgabe wirklich in einen Loop gepasst?
Beispiel-Runde · so klingt eine disziplinierte Session
Beispiel-Runde: Statusbericht im EPEC-Loop
User
„Statusbericht steht an — ich fang einfach an zu schreiben.“

Stimmt. liest … Es gibt drei Meeting-Notizen und ein altes Status-Doc — aber keinen roten Faden.

„1) Was ist fertig · 2) Was hängt · 3) nächste Schritte.“

„Zwei Termine waren veraltet. Korrigiert.“

Die Aufgabe
Statusbericht — aus verstreuten Notizen einen klaren Wochenbericht machen.
Das Learning
Der Navigator hat keine Zeile getippt — und trotzdem die Qualität bestimmt. Jedes der 4 Tore hat eine Abkürzung verhindert: kein Drauflos, kein Plan nur im Kopf, kein ungeprüftes „wird schon stimmen“, kein verlorenes Ergebnis. Das ist Pairing mit dem Agenten: Der Mensch hält die Tore, das Modell rennt.
