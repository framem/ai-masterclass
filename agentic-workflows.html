Warum überhaupt ein Workflow?

Ein Prompt erzeugt Text.
Ein Workflow erzeugt Ergebnisse.

Sobald ein Modell handeln soll — Dateien lesen, Code ändern, Tests laufen lassen — reicht der einzelne Prompt nicht mehr. Es braucht eine wiederholbare Choreografie: was geschieht in welcher Phase, mit welcher Information, unter welcher Kontrolle.

Naiv · One-Shot
„Mach das halt."
unstrukturiert
user  „Bau mir Feature X."
agent  schreibt sofort Code, ohne den Rest zu kennen
  • erfindet Konventionen, statt sie zu lesen
  • duplikate Funktionen, Stilbrüche
  • kein Punkt, an dem ein Mensch eingreifen kann
Strukturiert · Workflow
Erst verstehen,
dann handeln.
phasiert
01  Code & Kontext lesen
02  Plan vorschlagen, freigeben lassen
03  klein implementieren, prüfen
04  sauber committen
  • jede Phase erzeugt etwas Prüfbares
  • Mensch greift gezielt an Übergängen ein
  • wiederholbar über Teams und Aufgaben hinweg
Merksatz
Agentisch arbeiten heißt nicht „Agent macht alles allein". Es heißt: klare Phasen, klare Übergaben — der Agent füllt die Phasen, der Mensch öffnet die Tore.

Warum gerade diese vier Phasen?

Jede Phase schützt vor einem
konkreten Fehlermuster.

Wer eine Phase überspringt, bekommt zuverlässig den dazugehörigen Fehler. Deshalb ist die Reihenfolge selbst die wichtigste Schutzmaßnahme.

Halluzinierten Konventionen

Ohne Lesen erfindet der Agent Pfade, Namen und Patterns, die im Repo gar nicht existieren — Code, der lokal nach Lehrbuch aussieht und im Projekt komplett aus dem Rahmen fällt.

Falsche Aufgabe gelöst

Ohne expliziten Plan wird der erste Aspekt der Frage gelöst, der dem Modell ins Auge sticht — selten der wichtigste. Der Plan macht Annahmen sichtbar, bevor Code sie zementiert.

// Anti-Pattern:
500 Zeilen Code zur Frage, ob es überhaupt der richtige Ort ist.

schützt vor →
Theorie ohne Realität

Pläne, die nie laufen, sind nie falsifizierbar. Erst die Ausführung — Schritt für Schritt, mit Tests dazwischen — zeigt, was wirklich greift. Kleine Iterationen schlagen den großen Wurf.

// Anti-Pattern:
„Im Prinzip funktioniert das so…" — ohne npm test.

Verlorener Arbeit

Ohne Commit ist nichts reproduzierbar, nichts review-fähig, nichts wieder einzuspielen, wenn der nächste Versuch schiefgeht. Der Commit gibt dem Menschen den natürlichen Stop-and-Check-Punkt.

Was den Workflow am Leben hält

Drei Hebel, drei Stolperfallen.

+ Hebel
Was den Workflow trägt
Aufgaben klein schneiden

Ein Workflow pro Outcome, nicht pro Feature-Epic. Lieber zehn kleine Loops als ein heroischer.

Plan explizit absegnen

Das Gate zwischen Plan und Code ist der billigste Punkt, um eine falsche Richtung zu stoppen.

Häufig committen

Jeder grüne Zwischenstand ist ein Speicherpunkt. Ein Commit alle 20 Minuten ist normal, kein Overhead.

− Stolperfallen
Was den Workflow zerbricht
Plan im Kopf des Modells lassen

Ein Plan, der nicht aufgeschrieben ist, ist kein Plan. Er kann nicht geprüft, nicht verbessert, nicht erinnert werden.

Mitten in Execute die Aufgabe ändern

Neue Anforderung = zurück zu Plan. Nicht im laufenden Loop draufsatteln — der Context wird sonst zur Suppe.

Den Agent „einfach machen" lassen

Ohne Gate kein Workflow — nur ein langer Monolog mit Schreibrechten.

Take-away
Explore → Plan → Execute → Commit ist kein Prozess-Overhead, sondern ein Vertrag: der Agent verspricht, in dieser Reihenfolge zu arbeiten — der Mensch verspricht, an den richtigen Stellen einzugreifen.

EPEC an die Aufgabe anpassen

Ein Grundmuster, drei Spielarten.

Das Skelett bleibt gleich. Was sich ändert, ist das Artefakt, das den Inner-Loop steuert: ein roter Test, ein Zielbild — oder der Plan selbst.

VARIANTE 01
verifizierbar
TDD-Loop
der Test ist das Ziel
Explore → Plan →
Tests schreiben (rot) → Commit →
Code, bis grün → Commit

Erst Tests schreiben, die bewusst fehlschlagen — und committen. Dann implementieren, bis alles grün ist. Der Agent jagt ein hartes, prüfbares Ziel statt einer vagen Beschreibung.

Wann
Logik mit klaren Erwartungen · Bugfixes (Regression!) · Refactorings
VARIANTE 02
sichtbar
Visual-Loop
das Zielbild ist das Ziel
Explore → Plan →
Code → Screenshot → vergleichen →
Code → … → Commit

Mock oder Screenshot als Zielbild mitgeben. Der Agent baut, macht selbst einen Screenshot, vergleicht — und iteriert. 2–3 Runden sind normal, die erste Version ist selten die beste.

Wann
Frontends · Reports · Diagramme — alles mit sichtbarem Output
VARIANTE 03
volle Kontrolle
Plan-only
der Mensch übernimmt Execute
Explore → Plan →
✋ Übergabe an den Menschen →
selbst implementieren

Der Agent liest und plant — getippt wird selbst. Die billigste Variante, fremden Code zu verstehen, und die sicherste, wenn jede Zeile zählt.

Wann
Security-kritischer Code · Onboarding in fremde Codebases · Architektur-Fragen
Merksatz
Die Reihenfolge ist nie verhandelbar — das Artefakt schon. Gib dem Agenten das härteste Ziel, das die Aufgabe hergibt: Test > Zielbild > Beschreibung.

Wenn der Loop entgleist

Vier Stufen der Kurskorrektur.

Agenten laufen nicht fehlerfrei — das ist eingeplant. Entscheidend ist, früh und billig zu korrigieren, statt zuzuschauen. Von sanft nach hart:

01
Unterbrechen
Signal: falsche Richtung, aber nah dran
Stoppen (Esc), Anweisung nachschärfen, weiterlaufen lassen — der Kontext bleibt erhalten.
kostet: Sekunden
02
Zurückspulen
Signal: der Prompt selbst war das Problem
Zum letzten Checkpoint zurück, Prompt präziser formulieren, neu ansetzen — statt auf einer schiefen Antwort weiterzubauen.
kostet: Minuten
03
Kontext leeren
Signal: die Session ist zur Suppe geworden
Stand kurz zusammenfassen lassen → /clear → mit der Zusammenfassung frisch starten. Alte Irrwege vergiften sonst jede neue Antwort.
kostet: einen Neustart
04
Hard Reset
Signal: der Code selbst ist verbaut
git checkout zum letzten grünen Commit, neuen Loop bei Plan beginnen — nicht im Trümmerfeld weiterflicken.
kostet: einen Loop
Heuristik
Zweimal nachgebessert und immer noch falsch? Eine Stufe eskalieren. Sunk-Cost gilt auch für Prompts — und häufige Commits sind der Grund, warum Stufe 4 billig ist.

Dem Agenten geben — oder selbst machen?

Entscheide pro Aufgabe: voller Agenten-Loop, nur planen lassen — oder gar nicht delegieren.

Agent · voller Loop
klar geschnitten · prüfbares Ziel · Muster vorhanden
Plan-only
Agent liest & plant · Mensch führt aus
Selbst machen
Urteil & Verantwortung · oder schlicht schneller
„Flaky Test reparieren: OrderService.test schlägt jeden 5. CI-Run fehl — Repro-Schritte sind dokumentiert."
✓ Agent — klar geschnitten, Repro vorhanden, Erfolg prüfbar: Test läuft 20× grün.
„GET /invoices um Pagination erweitern — das Muster existiert bereits in GET /orders."
✓ Agent — Vorbild im Code ist das halbe Briefing. Explore findet es, Execute kopiert das Muster.
„Codebase geerbt: verstehen, wie der Checkout-Flow funktioniert — bevor ihr ihn anfasst."
✓ Plan-only — Explore + Erklärung sind Gold, aber noch nichts ändern lassen. Onboarding ist Lese-Arbeit.
„Token-Handling im Auth-Service umbauen — security-kritisch, jede Zeile muss verstanden sein."
✓ Plan-only — Analyse und Plan vom Agenten, getippt wird selbst. Hohes Risiko heißt: Mensch schreibt jede Zeile.
„Entscheiden, ob das Team auf Microservices migriert — strategisch, bindet zwei Quartale."
✓ Selbst — kein prüfbares Ziel, volle Verantwortung. Input sammeln lassen ist ok — die Entscheidung ist deine.
„Tippfehler in einer Log-Meldung korrigieren — du weißt genau, welche Datei und Zeile."
✓ Selbst — wenn du die Lösung schon im Kopf hast, ist Tippen schneller als Briefen. Delegation hat Overhead.

Echte Praxis — Laptops auf, Disziplin an

Macht einen EPEC-Loop — zu zweit, mit echter Disziplin.

15 Min Coden · 5 Min Debrief. Einer schreibt, einer hält das Brett — und schiebt den Agenten zurück, wenn er ein Tor überspringen will.

Rolle · Driver
Driver
am Laptop · tippt
Öffnet ein Repo + Coding-Agent in der IDE. Sagt laut, was passiert. Hat kein Recht, eigenmächtig Phasen zu überspringen.
Rolle · Navigator
Navigator
EPEC-Polizei · hält das Brett
Beobachtet, fragt, blockt. Darf nicht selbst tippen. Setzt die 4 Tore rechts durch — ist die wichtigste Person im Pair.

Das Learning
