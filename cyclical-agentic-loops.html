„Why Linear AI Chains Are Dead"

Der Loop ist die Recovery-Strategie, die Chains nie hatten.

Beispiel aus dem Artikel: „Hol den Apple-Aktienkurs, fasse die News zusammen, schreib einen Tweet." Drei Schritte, eine Frage — was passiert, wenn Schritt 2 ein Rate-Limit fängt?

Chain · was passiert
„Sorry, das hat nicht geklappt."
01 Kurs holen → ok
02 News-API → 429 rate-limited
03 Tweet schreiben
→ Pipeline crasht. User bekommt einen Error.
Brittle
Linear ist gut für deterministische Transformationen — und schlecht für alles, was schiefgehen kann.
Loop · was passiert
„Rate-Limit — ich versuch's mit Cache."
act News-API aufrufen
obs 429 — Tool-Fehler im State protokollieren
reason „API blockiert — nimm Cache + warne im Tweet."
act cachedNews(7d) aufrufen → ok
→ Tweet gepostet. Nutzer merkt nichts.
Resilient
Der Loop sieht den Fehler im State und reagiert — wie ein Mensch beim zweiten Versuch.

Was steckt im Loop?

Drei Phasen, vier Bauteile — mehr nicht.

ReAct (Reason + Act) ist nicht nur ein Prompt-Trick. Es ist eine Zustandsmaschine: das Modell denkt, ruft ein Tool, sieht das Ergebnis, denkt nochmal — bis ein Stopp-Kriterium hält.

Phase 01 · Thought
Reason
Modell liest den State, formuliert den nächsten Schritt.
→
Phase 02 · Action
Act
Tool-Aufruf — eine deterministische, isolierte Funktion.
→
Phase 03 · Observation
Observe
Ergebnis (oder Fehler) landet im State — und triggert Reason neu.
↺ back to Reason
Die 4 Bauteile · Microservice-Analogie aus dem Artikel
Bauteil 01
Model
das Gehirn · API-Gateway
Interpretiert Intent, routet Aufrufe. Hält selbst keine Daten.
Bauteil 02
Tools
die Microservices
Zustandslose Funktionen — fetchen, rechnen, schreiben. Wissen nichts vom Workflow.
Bauteil 03
Memory
der Shared State
Mutables Objekt, das den Loop überlebt. Ohne State → Agent mit Amnesie.
Bauteil 04
Orchestration
der Event-Bus
Definiert die Edges — wann gehst du wohin? Wann zurück? Wann raus?
Merksatz
Wenn du einen Loop entwirfst, beantwortest du vier Fragen: Was ist der State? Was sind die Nodes? Wann loopst du zurück? Und wann brichst du ab?

Wenn der Loop nicht stoppt

Dein Agent dreht im Kreis. Was baust du ein?

Szenario
Dein Ticket-Klassifizierer hat einen reflectOnPriority-Node, der bei unscharfen Tickets immer wieder „nochmal analysieren" sagt. Bei einem mehrdeutigen Ticket pendelt er endlos zwischen High und Urgent. Vercel killt deine Function nach 10 Sekunden. Was tust du?
Auflösung · der Drei-Schichten-Notausgang
Ein guter Loop hat immer mehrere Stoppgründe.
Der Artikel nennt das die Max Iteration Policy: ein Zähler im State, eine Conditional Edge, ein __end__ bei Überschreitung. In Produktion kommen meist noch zwei weitere Schichten dazu: Timeout (Wall-Clock, gerade bei Vercel/Lambda kritisch) und Token/Cost-Cap. Drei Notausgänge — gerne alle drei gleichzeitig.
SCHICHT 01
Max-Iter · Counter im State, Cap z.B. 3–10
SCHICHT 02
Timeout · Wall-Clock, kürzer als die Function-Timeout
SCHICHT 03
Cost-Cap · Tokens/€ pro Run, harter Stopp
Wähle die Karte — Auflösung erscheint darunter.

Vom einzelnen Loop zum Team

Aus einem Loop wird ein Team.

Ein Loop ist ein Agent, der denkt. Doch ein einzelner Loop stößt an Grenzen: begrenzter Context, breites Spezialwissen schwer, alles seriell. Die Antwort ist Spezialisierung — mehrere Loops mit eigener Rolle, eigenen Tools, oft eigenem Modell. Wer die Orchestration aus dem Anatomie-Kapitel eine Ebene höher zieht, landet hier: bei Topologien und dem Protokoll A2A.

Topologien
Drei Wege, Loops zu einem Team zu verschalten
SUPERVISOR
Orchestrator zerlegt & delegiert, sammelt Ergebnisse ein.
PIPELINE
Reihenschaltung — Output von A ist Input von B.
SCHWARM
Peer-to-Peer — jeder kann jeden ansprechen, kein Chef.
Beispiel · Recherchebericht
Supervisor teilt auf in: Recherche-Agent (Web + PDFs) → Analyse-Agent (Zahlen, Code) → Schreib-Agent (Bericht) → zurück.
Protokoll
A2A — Loops finden sich
offen · 2025

Agent-to-Agent ist das, was MCP für Tools ist — nur eine Ebene höher: ein Standard, mit dem Loops von verschiedenen Anbietern sich gegenseitig Aufgaben stellen, Status verfolgen, Ergebnisse austauschen. Jeder Loop bleibt für die anderen eine Black Box.

Agent Card — öffentliches Profil: was kann ich, wo erreichbar.
Task — strukturierte Anfrage mit Zustand & Artefakten.
Merksatz
MCP vs A2A — komplementär:  MCP verbindet Loop ↔ Werkzeug.  ·   A2A verbindet Loop ↔ Loop.  Ein produktives System nutzt meist beides.

Mitnehmen

Sechs Fragen, ein solider Loop.

Stelle sie in dieser Reihenfolge. Wer die ersten zwei nicht beantworten kann, baut keinen Loop — baut eine Endlosschleife.

Der Loop
Reason Act Observe Agent
FRAGE 01
Lohnt sich der Loop?
Adaptiv? Fehler-Recovery? Re-Plan nötig? → Loop.
Deterministisch & einmalig? → Chain.
FRAGE 02
Was ist der State?
Was muss zwischen Iterationen erhalten bleiben? History, Counter, Zwischenergebnisse — typisierte Felder, kein any.
FRAGE 03
Was sind die Nodes?
Reason / Act / Observe sauber trennen. Ein Node = eine Verantwortung. Nicht „Modell denkt UND ruft Tool" im selben Schritt.
FRAGE 04
Wann loopst du zurück?
Conditional Edge, klar formulierbar in einem Satz: „Wenn ___, dann zurück zu ___".
FRAGE 05
Wann hörst du auf?
Erfolgs-Kriterium vor dem Bau definieren. Sonst weißt du nie, ob der Loop fertig ist oder nur müde.
FRAGE 06
Wo ist der Notausgang?
Drei Schichten: Max-Iter · Timeout · Cost-Cap. Lieber zwei zu viel als einen zu wenig.
Und wenn ein Loop nicht reicht?
Dann spezialisieren: mehrere Loops als Team — Supervisor · Pipeline · Schwarm, verbunden über A2A (siehe Kapitel 08). Erst nehmen, wenn Spezialisierung den Mehraufwand schlägt.
Merksatz
Ein Loop ist kein Hack — er ist die Architektur der Selbstkorrektur. Wer Chains baut, baut Hoffnung. Wer Loops baut, baut Recovery.
