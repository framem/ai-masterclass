Was in jedem Subagenten passiert, wenn er ein Tool ruft

Der Tool-Use-Loop.

Ob Single-Agent oder ein einzelner Worker im Team — der Mechanismus ist derselbe. Jeder Subagent läuft seinen eigenen Loop: Du registrierst Tools (Name + JSON-Schema), das Modell antwortet mit einem tool_use-Block, dein Code führt das Tool aus, du gibst das Ergebnis zurück. Wiederholen.

SCHRITT 01
Tools registrieren
Du gibst dem Modell pro Tool: Name, Beschreibung, Input-Schema (JSON).
tools = [{
  name: "search_jira",
  description: "Suche Tickets nach JQL.",
  input_schema: {…}
}]
SCHRITT 02
Modell fragt an
Statt Text gibt das Modell ein tool_use-Objekt zurück — mit Tool-Name und Argumenten.
{
  type: "tool_use",
  name: "search_jira",
  input: { jql: "P1 AND…" }
}
SCHRITT 03
Dein Code führt aus
Dein Code (oder MCP-Server) führt das Tool tatsächlich aus — das Modell kann nicht selbst auf die API.
result = jira.search(
  jql=input["jql"]
)
SCHRITT 04
Ergebnis zurück
Das Ergebnis fließt zurück ins Modell als tool_result. Modell denkt weiter oder antwortet final.
{
  type: "tool_result",
  content: [...]
}
Wichtig 01
Das Modell führt nichts aus. Es schreibt nur, was ausgeführt werden soll. Sicherheit/Authorisation liegen bei deinem Code.
Wichtig 02
Tool-Beschreibungen sind Prompts. Schlechte Beschreibungen = schlechte Tool-Wahl. Hier zählt jedes Wort.
Wichtig 03
Loop endet, wenn das Modell stop_reason: end_turn liefert — oder dein Code einen Loop-Limit reißt.
Merksatz
Ein „agentic Tool-Call" ist kein magischer Aufruf — es ist ein JSON-Block hin, ein JSON-Block zurück. Wer das versteht, debugged einzelne Agenten und ganze Teams.

Der USB-C-Stecker für LLM-Tools

MCP — ein Protokoll, viele Quellen.

Model Context Protocol: offener Standard, der LLM-Clients (Desktop-App, IDE, Agent) eine einheitliche Sprache gibt, um Tools, Daten und Ressourcen einzubinden. Pro Datenquelle ein MCP-Server — jeder MCP-fähige Client kann ihn nutzen. Im Team heißt das: jeder Subagent bekommt genau die Server, die seine Rolle braucht.

Der Trick
Ein Server pro Quelle — geschrieben einmal, genutzt von allen Clients. Keine N×M-Integrationen mehr.
Was ein MCP-Server exponiert
tools
Aktionen (lesen, schreiben, suchen) — vom Modell aufrufbar.
resources
Daten/Dateien, die als Kontext geladen werden können.
prompts
Vorgefertigte Templates, die der Client einbinden kann.
sampling
Optional: Server bittet Client um Modell-Aufrufe.
Ohne MCP vs. mit MCP
N × M Integrationen
Pro Client × Tool eine eigene Anbindung. Jede API-Änderung trifft alle.
N + M Integrationen
Jeder Client spricht MCP. Jedes Tool stellt einen Server. Patch im Server — alle profitieren.
Architektur-Frage
Wenn dein Team einen eigenen Connector baut, frag zuerst: Gibt es einen MCP-Server dafür? Falls nein — schreibt euren als MCP-Server, nicht als Custom-Tool. Andere Teams (und Vendor-Clients) profitieren mit.

Konkret: Jira & Confluence per MCP

So sieht MCP im Alltag aus.

Ein MCP-Server kapselt die jeweilige API als standardisierte Tools. Ein Subagent muss weder Atlassian-REST-Endpoints noch Auth-Details kennen — er sagt, was er will, der Server übersetzt.

Im Team
Mehrere Server,

Merksatz
Die API gehört dem MCP-Server, nicht dem Agenten. Atlassian patcht ihre REST-Schnittstelle — der Server wird aktualisiert, alle MCP-fähigen Clients profitieren mit. Keine Doppelarbeit pro Tool.

Zu zweit. Ihr nehmt eine Aufgabe und zerlegt sie in ein Team: Welche Subagenten braucht es? Für jeden: Rolle, Input, Tools, Rückgabe. Das Learning
Der Team-Kritiker hat kein Team entworfen — er hat bei jeder Rolle nur gefragt: „Braucht das wirklich einen eigenen Agenten?“ Das strich eine Rolle (Formatierer → Orchestrator) und beschränkte den Daten-Agenten auf read-only.
