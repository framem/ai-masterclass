Kein Schalter — ein Schieberegler

Wer trifft die Entscheidungen?

Der Unterschied liegt nicht in den Modellen, sondern in der Kontrolle des Kontrollflusses. Bei einem Workflow entscheidet dein Code, was als nächstes passiert. Bei einem Agenten entscheidet das Modell. Dazwischen liegt alles.

STATION 01
einfachster Fall
Augmented LLM
ein Modell + Werkzeuge

Ein Aufruf, optional Retrieval/Tools/Memory. Kein Loop, keine Verzweigung — Aufruf rein, Antwort raus.

# Beispiel
chat(prompt + RAG-context)
↑ Kontrolle
100 % beim Aufrufer. Das Modell antwortet — sonst nichts.
STATION 02
vorhersehbar
Workflow
Code orchestriert mehrere Modell-Aufrufe

Dein Code legt fest, in welcher Reihenfolge welche Modelle/Tools laufen. Verzweigungen sind hartcodiert, keine offene Schleife.

# Beispiel
classify → if A: extract
    else: summarize
↑ Kontrolle
Beim Code. Reproduzierbar, testbar — dafür starrer.
STATION 03
offen
Agent
Modell wählt Tools & Reihenfolge selbst

Loop: Modell denkt → ruft Tool → liest Ergebnis → entscheidet selbst, was als nächstes — bis ein Abbruchkriterium greift.

# Beispiel
while not done:
  think + call_tool()
↑ Kontrolle
Beim Modell. Flexibel — und teurer, langsamer, schwerer zu testen.
Merksatz
Bei einem Workflow ist der Pfad bekannt, das Modell füllt die Schritte. Bei einem Agenten ist nur das Ziel bekannt, das Modell findet den Pfad. Beides ist erlaubt — aber kosten unterschiedlich.

Wann was — die Architektur-Frage

Workflow oder Agent?

Default: so wenig Autonomie wie möglich. Jeder Schritt in Richtung Agent erkauft Flexibilität mit Kosten, Latenz und Unvorhersehbarkeit. Diese Tabelle hilft bei der Wahl.

Dimension Workflow Agent
Pfad bekannt? ja — fest verdrahtet nein — Modell entscheidet
Reproduzierbarkeit hoch — gleiche Eingabe, gleicher Pfad niedrig — Pfad variiert mit Sample
Testbarkeit Unit-Tests pro Schritt End-to-End-Evals, Trajektorien
Kosten / Latenz vorhersagbar, niedrig unbegrenzte Loops möglich
Aufgaben-Form stabil, wiederkehrend offen, explorativ
Fehlerbild starr — übersehene Ausnahmen Driftet, irrt, dreht im Kreis
→ Default
Workflow

Augmented LLM oder ein Pattern aus der vorherigen Folie. 90 % der Produktionsfälle.

→ Eskalation
Agent

Wenn Aufgaben genuin offen sind und die Anzahl der Schritte nicht vorab bekannt — z. B. agentic Coding, Research, Exploration.

→ Pflicht für Agenten
  • Loop-Limit + Token-Budget
  • klar definierte Stop-Conditions
  • Mensch-im-Loop bei irreversiblen Aktionen
  • Trace + Replay pro Run
Faustregel
Workflow → Vorhersagbarkeit. Agent → Flexibilität. Wähle die einfachste Lösung, die das Problem löst. Komplexität ist eine Schuld, die du später bezahlst — in Bugs, Kosten und Vertrauen.

Welches Muster passt? Eine richtige Antwort pro Szenario.

„Outline → Entwurf → Lektorat → Veröffentlichen." Jeder Schritt nimmt das Ergebnis des vorigen — kein Verzweigen, kein Loop.
✓ Prompt-Chaining — linearer Output-zu-Input-Chain, vorab festgelegt. Routing würde verzweigen, ein Agent den Pfad selbst wählen.
„Pro Support-Ticket: einordnen (Bug / Frage / Refund), je nach Kategorie passend behandeln — bei Bug an Eng eskalieren."
✓ Routing — Klassifikation + Dispatch. Der Klassifizierer ist die einzige variable Stelle, jede Sub-Pipeline bleibt deterministisch.
„Generiere einen Blogpost. Ein Editor-Modell bewertet Stil & Faktendichte — solange Score < 8, überarbeite."
✓ Evaluator-Optimizer — Generieren + Bewerten + Verfeinern bis zur Qualitätsschwelle. Der definierende Feedback-Loop, mit endlicher Iteration.
„Ein Dokument gleichzeitig auf Sentiment, Thema und Sprache analysieren, dann die Ergebnisse zusammenführen."
✓ Parallelisierung — unabhängige Teilaufgaben im Fan-out, danach Aggregat. Routing wählt einen Pfad, hier laufen mehrere gleichzeitig.
„Benenne eine Funktion im ganzen Repo um." Ein Modell findet zur Laufzeit die betroffenen Dateien, verteilt pro Datei eine Teil-Änderung an Worker-Modelle und führt alles zusammen.
✓ Orchestrator-Workers — ein Modell zerlegt die Aufgabe zur Laufzeit und delegiert an Worker. Parallelisierung hätte einen festen, vorab definierten Split; ein Agent würde auch das Ziel selbst bestimmen.
„Plan ein langes Wochenende in Lissabon — 600 € Budget, Kunst-Fokus." Recherchier Flüge, Hotels, Museen — und koordinier alles.
✓ Agent — die Schritte hängen davon ab, was du findest. Der Pfad lässt sich nicht vorab planen — genau die Definition für einen Agenten.

Was passiert, wenn ein Modell ein Tool ruft

Der Tool-Use-Loop.

Egal ob Workflow oder Agent — der Mechanismus ist derselbe. Du registrierst Tools (Name + JSON-Schema), das Modell antwortet mit einem tool_use-Block, dein Code führt das Tool aus, du gibst das Ergebnis zurück. Wiederholen.

SCHRITT 01
Tools registrieren
Du gibst dem Modell pro Tool: Name, Beschreibung, Input-Schema (JSON).
tools = [{
  name: "search_jira",
  description: "Suche Tickets nach JQL.",
  input_schema: {…}
}]
SCHRITT 02
Modell fragt an
Statt Text gibt das Modell ein tool_use-Objekt zurück — mit Tool-Name und Argumenten.
{
  type: "tool_use",
  name: "search_jira",
  input: { jql: "P1 AND…" }
}
SCHRITT 03
Dein Code führt aus
Dein Code (oder MCP-Server) führt das Tool tatsächlich aus — das Modell kann nicht selbst auf die API.
result = jira.search(
  jql=input["jql"]
)
SCHRITT 04
Ergebnis zurück
Das Ergebnis fließt zurück ins Modell als tool_result. Modell denkt weiter oder antwortet final.
{
  type: "tool_result",
  content: [...]
}
Wichtig 01
Das Modell führt nichts aus. Es schreibt nur, was ausgeführt werden soll. Sicherheit/Authorisation liegen bei deinem Code.
Wichtig 02
Tool-Beschreibungen sind Prompts. Schlechte Beschreibungen = schlechte Tool-Wahl. Hier zählt jedes Wort.
Wichtig 03
Loop endet, wenn das Modell stop_reason: end_turn liefert — oder dein Code einen Loop-Limit reißt.
Merksatz
Ein „agentic Tool-Call" ist kein magischer Aufruf — es ist ein JSON-Block hin, ein JSON-Block zurück. Wer das versteht, debugged Agenten.

Der USB-C-Stecker für LLM-Tools

MCP — ein Protokoll, viele Quellen.

Model Context Protocol: offener Standard, der LLM-Clients (Desktop-App, IDE, Agent) eine einheitliche Sprache gibt, um Tools, Daten und Ressourcen einzubinden. Pro Datenquelle ein MCP-Server — jeder MCP-fähige Client kann ihn nutzen.

Der Trick
Ein Server pro Quelle — geschrieben einmal, genutzt von allen Clients. Keine N×M-Integrationen mehr.
Was ein MCP-Server exponiert
tools
Aktionen (lesen, schreiben, suchen) — vom Modell aufrufbar.
resources
Daten/Dateien, die als Kontext geladen werden können.
prompts
Vorgefertigte Templates, die der Client einbinden kann.
sampling
Optional: Server bittet Client um Modell-Aufrufe.
Ohne MCP vs. mit MCP
N × M Integrationen
Pro Client × Tool eine eigene Anbindung. Jede API-Änderung trifft alle.
N + M Integrationen
Jeder Client spricht MCP. Jedes Tool stellt einen Server. Patch im Server — alle profitieren.
Architektur-Frage
Wenn dein Team einen eigenen Connector baut, frag zuerst: Gibt es einen MCP-Server dafür? Falls nein — schreibt euren als MCP-Server, nicht als Custom-Tool. Andere Teams (und Vendor-Clients) profitieren mit.

Konkret: Jira & Confluence per MCP

So sieht MCP im Alltag aus.

Ein MCP-Server kapselt die jeweilige API als standardisierte Tools. Der Agent muss weder Atlassian-REST-Endpoints noch Auth-Details kennen — er sagt, was er will, der Server übersetzt.

Merksatz
Die API gehört dem MCP-Server, nicht dem Agenten. Atlassian patcht ihre REST-Schnittstelle — der Server wird aktualisiert, alle MCP-fähigen Clients profitieren mit. Keine Doppelarbeit pro Tool.

Spickzettel · Woher bekommt das Modell den Wert?
A
Nutzer-Anfrage
Steht schon wörtlich im Auftrag — „leg ein Ticket für den Login-Bug an“.
B
Rückfrage
Fehlt und ist wichtig → das Modell muss nachfragen, statt zu raten. Die Beschreibung sollte das sagen.
C
Session / Kontext
Ergibt sich aus dem bisherigen Gespräch, angehängten Dateien oder dem geöffneten Projekt.
D
Anderer Tool-Call
Ein Lookup-Tool liefert vorher die ID — erst search_x, dann update_x.
E
Config / Default
Fest hinterlegt (Profil, Projekt, Umgebung) — gehört dann oft gar nicht ins Schema.
Tool-Ideen · pickt eine — oder nehmt eine echte aus eurem Alltag
create_ticket

trigger_deploy
Deployment einer Umgebung anstoßen.

So füllt das Modell die Parameter — drei Fälle
„Wie wird das Wetter morgen in Hamburg?“
→ get_weather( ort: "Hamburg" A, tag: "morgen" A ) — alles steht im Auftrag.
„Wie wird das Wetter?“
→ kein Tool-Call. Erst Rückfrage B: „Für welchen Ort?“ — der Pflicht-Parameter fehlt, raten verboten.
„Wie wird das Wetter in Kiel?“
→ get_weather( ort: "Kiel" A ) — tag weggelassen, Default greift: heute E.

Das Learning
Der Modell-Anwalt hat kein Schema geschrieben — er hat bei jedem Wert nur gefragt: „Woher weiß ich das?“ Das deckte zwei Dinge auf: eine Beschreibung, die Nachfragen statt Raten erzwingt, und ein Feld (Zugangsschlüssel), das gar nicht ins Schema gehört. Gutes Tool-Design entscheidet sich nicht am Parameter-Typ — sondern an der Quelle jedes Werts.

Wenn / Dann · drei Architektur-Heuristiken
Wenn jemand „Agent" sagt
Frag, welcher Schritt nicht vorab bekannt ist. Wenn keiner — Workflow.
Wenn der Loop > 10 Tool-Aufrufe braucht
Zerlege. Orchestrator + Workers oder Sub-Workflow — keine 50-Step-Agents.
Wenn ein Tool irreversibel ist
Mensch-im-Loop — Approval-Gate vor dem Aufruf, nicht danach.
Mitnehmen
Wähl die einfachste Architektur, die funktioniert: Workflow vor Agent, Agent nur bei genuin offenem Pfad — und Mensch-im-Loop, sobald ein Schritt irreversibel ist.
